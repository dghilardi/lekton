# RAG Evaluation

Lekton provides two CLI tools for measuring retrieval quality in the RAG pipeline:

| Binary | When to use it |
|--------|----------------|
| `rag-eval` | Evaluate one configuration against an already populated Qdrant collection. Fast, no ingest. |
| `rag-bench` | Compare multiple configurations in one run. For each config, it starts from a clean collection, ingests all documents, runs the queries, and writes individual plus comparative reports. |

This guide focuses on `rag-bench`, because it is the right tool for deciding how to tune the system. Use `rag-eval` for targeted debugging of a live configuration.

---

## 1. How Evaluation Works

For each query in the evaluation set, the pipeline returns:

- **pre-rerank**: the top-k chunks from the vector store, and from RRF fusion with Meilisearch when hybrid search is enabled.
- **post-rerank**: the same chunks after the cross-encoder pass, when a reranker is configured.

Three metrics are calculated for both result sets:

| Metric | Range | Meaning |
|--------|-------|---------|
| **Recall@k** | `[0, 1]` | Fraction of expected documents/fragments found in the first `k` results. If you expect 2 slugs and retrieve only 1, recall is 0.5. |
| **MRR** (Mean Reciprocal Rank) | `[0, 1]` | `1 / position of the first relevant result`. Rewards relevant results near the top. |
| **nDCG@k** | `[0, 1]` | Normalized Discounted Cumulative Gain. Rewards retrieving more relevant results and ranking them higher. |

All metrics are fractions from 0 to 1. A score of 1.0 is the maximum.

**Ground truth.** Each evaluation query declares what it expects. A chunk is considered relevant when at least one of these conditions is true:

- the chunk `document_slug` is listed in `expected_doc_slugs`;
- the chunk `point_id` is listed in `expected_chunk_ids` (rarely recommended, because ids change on every reindex);
- the chunk `chunk_text` contains at least one `expected_text_fragments` value using case-insensitive substring matching.

Text fragments are especially useful when you need to verify retrieval of the right paragraph, not just the right document.

## 2. Quick Start

### Prerequisites

- A reachable Qdrant instance (default `http://localhost:6334`).
- An OpenAI-compatible embedding service (Ollama, vLLM, OpenAI, etc.).
- Optional: Meilisearch for hybrid search tests; a Jina/Infinity-compatible reranker.

The development `docker compose` stack already starts Qdrant and Meilisearch.

### Directory Layout

```text
eval/
|-- configs/                   # One self-contained TOML config per variant
|   |-- baseline.toml
|   |-- smaller-chunks.toml
|   |-- larger-chunks.toml
|   |-- hybrid-rrf.toml
|   `-- expand-to-parent.toml
|-- docs/                      # Markdown documents ingested for the benchmark
|   |-- api-reference.md
|   |-- architecture.md
|   `-- ...
|-- queries.jsonl              # Query set with expected results
`-- reports/                   # Generated by rag-bench (gitignored)
    |-- baseline.json
    |-- hybrid-rrf.json
    `-- comparative.md
```

### Run the Benchmark

```sh
cargo run --bin rag-bench --features ssr --no-default-features -- --bench-dir eval
```

Primary stdout output, one row per config:

```text
baseline             post R@10=0.917  MRR=0.875  nDCG=0.901
smaller-chunks       post R@10=0.875  MRR=0.812  nDCG=0.847
hybrid-rrf           post R@10=0.958  MRR=0.917  nDCG=0.940
```

Detailed outputs:

- `eval/reports/<name>.json` for each config, including retrieved chunks, scores, and per-query metrics.
- `eval/reports/comparative.md` with aggregate tables and per-query detail.

### Useful Flags

| Flag | Default | When to use it |
|------|---------|----------------|
| `--bench-dir DIR` | `eval` | Keep multiple benchmark suites, for example `eval-prod` or `eval-internal`. |
| `--queries FILE` | `<bench-dir>/queries.jsonl` | Reuse the same config set with a different query corpus. |
| `--top-k N` | `10` | Stress the pipeline with fewer (`-k 3`) or more (`-k 20`) results. |
| `--only NAME[,NAME]` | all configs | Re-run only a subset after changing one config. |
| `--keep-collections` | `false` | Inspect Qdrant collections manually after the run. |

## 3. File Formats

### TOML Configs

Each file in `configs/` is self-contained: it includes all parameters required for one run and does not inherit from the server configuration. This prevents accidental drift between the evaluation environment and the current runtime config.

Required fields:

```toml
name                 = "baseline"            # short id used in filenames and collection names
description          = "..."                 # free text included in the comparative report

qdrant_url           = "http://localhost:6334"
embedding_url        = "http://localhost:11434/v1"
embedding_model      = "nomic-embed-text"
embedding_dimensions = 768
chunk_size_tokens    = 256
chunk_overlap_tokens = 64
```

Optional fields:

```toml
qdrant_collection     = "lekton_bench_xyz"   # default: lekton_bench_<name>
embedding_api_key     = ""
embedding_headers     = { x_producer = "lekton-bench" }

expand_to_parent      = false
hybrid_search_enabled = false                # requires meilisearch_url
meilisearch_url       = ""
meilisearch_api_key   = ""

reranker_url          = ""                   # empty means reranker disabled
reranker_model        = ""
reranker_api_key      = ""
reranker_headers      = {}

# Optional LLM steps for testing analyzer / HyDE
[analyzer]
model = "llama3.2:1b"
url   = "http://localhost:11434/v1"

[hyde]
model = "llama3.2:1b"
url   = "http://localhost:11434/v1"
```

**Validation.** The benchmark rejects configs with `chunk_size_tokens = 0`, `chunk_overlap_tokens >= chunk_size_tokens`, `hybrid_search_enabled = true` without `meilisearch_url`, and similar invalid combinations. See `BenchConfig::validate` for the full list.

### Documents

All `.md` files in `docs/` are ingested. The slug is the filename without extension (`getting-started.md` becomes `getting-started`). The title comes from the first `# H1`; when no H1 exists, the slug is used with hyphens replaced by spaces.

### Queries (`queries.jsonl`)

One JSON object per line. Empty lines and lines starting with `#` are ignored.

```jsonc
{
  "id": "Q01",                                      // optional: default Q001, Q002, ...
  "query": "How do I push docs from CI?",           // required
  "expected_doc_slugs": ["getting-started"],        // at least one of these three lists
  "expected_chunk_ids": [],                         // discouraged; ids change on reindex
  "expected_text_fragments": ["service_token"]      // case-insensitive substring match
}
```

At least one `expected_*` field must contain values, otherwise the benchmark rejects the query. Empty expectations would produce invalid metrics.

**How many records do I need?** A representative set of 30-50 queries is a reasonable minimum for a useful Recall@k / MRR / nDCG signal. Below 10 records, differences are usually dominated by noise.

## 4. Best Practices

### Build a Useful Evaluation Set

1. **Mix query types.** Include lookups (`Which endpoint does X?`), explanations (`How does X work?`), and cross-document questions (`Compare X and Y`).
2. **Write realistic user language.** Do not simply paraphrase document titles; embeddings will usually find those anyway. Use the wording your users would actually type.
3. **Use one paragraph-level expectation per query** when you need to guarantee the right paragraph was retrieved. Add a distinctive, low-ambiguity string to `expected_text_fragments`.
4. **Update the set when documents are renamed.** Slugs are the matching key: renaming `security-rbac` to `auth-and-rbac` will make every query expecting the old slug fail.
5. **Avoid `expected_chunk_ids`.** They survive only until the next reindex. Use them for chunking stability checks, not long-lived ground truth.

### Compare Configurations

- **Always compare against the baseline.** The useful question is not "is this config good?", but "is this config better than the baseline?".
- **Read the three metrics together.** High recall with low MRR means the right documents are present but ranked too low; a reranker may help. Low nDCG with high MRR means the first result is right but the rest are noisy; try smaller chunks or reranking.
- **Compare pre-rerank vs post-rerank.** If `pre` and `post` are roughly equal, the reranker is not adding value. Common causes are the wrong model, too small a top-k, or an eval set that is too easy.
- **Change one variable at a time.** If you change `chunk_size` and enable the reranker in the same config, you cannot attribute the delta. Benchmarks are cheaper than ambiguous results.

## 5. Starting Points by Corpus Type

These are practical starting points, not universal truths. Confirm them with a benchmark on your own query set.

### Short, Atomic Technical Docs

Examples: READMEs, runbooks, FAQs.

- `chunk_size_tokens = 128-256`, `chunk_overlap_tokens = 32-64`
- `expand_to_parent = false`, because the documents are already small
- Hybrid search is strongly recommended: these corpora often contain acronyms, environment variables, error codes, and CLI flags that embeddings blur.
- Reranking is useful mainly when you have more than 10 very similar documents.

### Long Architecture Documents

Examples: design docs, ADRs, RFCs.

- `chunk_size_tokens = 384-512`, `chunk_overlap_tokens = 96-128`
- `expand_to_parent = true`, because the LLM often needs the whole section, not just one paragraph.
- HyDE can help when queries are abstract compared with the source text.

### API Reference / OpenAPI Prose

Examples: endpoint descriptions, parameters, examples.

- `chunk_size_tokens = 256`, `chunk_overlap_tokens = 64`
- Hybrid search is strongly recommended: endpoint and field names are exact strings that full-text search often finds better than embeddings.
- Reranking has high ROI because short endpoint descriptions can look very similar to each other.

### Long Tutorials / How-To Guides

Examples: sequential steps and troubleshooting guides.

- `chunk_size_tokens = 512`, `chunk_overlap_tokens = 128`
- `expand_to_parent = true` when tutorials have clear sections such as `## Step 1` and `## Step 2`.
- The query analyzer can help when queries span multiple steps (`how do I do X after Y?`).

### Mixed Knowledge Bases

Examples: team pages, decision records, post-mortems.

- `chunk_size_tokens = 256`, `chunk_overlap_tokens = 64`
- Hybrid search is recommended for people, project, and service names.
- Tags and access levels matter: if benchmark documents have unrealistic metadata, a public-only benchmark will not reflect the real user experience.

## 6. Common Recipes

### Evaluate Whether to Add a Reranker

Create two configs that differ only by `reranker_url`:

```toml
# without-reranker.toml: name = "without-reranker", reranker_url = ""
# with-reranker.toml:    name = "with-reranker", reranker_url = "http://..."
```

Run `cargo run --bin rag-bench ...`. Compare the **post-rerank** columns. If the nDCG delta is below 2 percentage points, the reranker probably does not justify the latency and cost for that corpus.

### Find the Best Chunk Size

Create 4 configs with `chunk_size_tokens` in `{128, 256, 384, 512}` and `chunk_overlap_tokens = chunk_size / 4`. Run the benchmark. The nDCG vs chunk size curve shows the best starting point. Expect this to depend heavily on the average document length.

### Change Embedding Model

Create two configs that differ only by `embedding_model` and `embedding_dimensions`. Each config uses its own collection, derived from `name`, so there is no contamination between runs. Use the same query set for the comparison.

### Test Only New Queries

Keep `eval/queries.jsonl` as the historical set and create `eval/queries-new.jsonl` for new cases. Then run:

```sh
cargo run --bin rag-bench --features ssr --no-default-features -- \
    --bench-dir eval --queries eval/queries-new.jsonl --only baseline
```

## 7. Current Limits

- **Retrieval only, not answer quality.** The benchmark measures what the pipeline retrieves, not how good the final LLM answer is. Answer quality needs judge-based metrics such as faithfulness and answer relevance, which are out of scope for now.
- **No document noise model.** All documents are ingested with `access_level="public"` and `is_draft=false`. This does not evaluate RBAC filtering quality.
- **Hybrid search: only Qdrant is managed.** `rag-bench` reindexes Qdrant for every config but does not populate Meilisearch. To test hybrid mode seriously, use a Meilisearch instance already aligned with `eval/docs/`; the simplest path is a development Lekton server previously loaded through `demo/load_demo.sh`. If Meilisearch is empty or out of sync, the pipeline falls back to vector-only retrieval and emits a warning.
- **Manual evaluation set.** There is no automatic query generation from documents. For large corpora, tools such as [RAGAS](https://github.com/explodinggradients/ragas) can complement this workflow.
- **Determinism depends on the backend.** Embeddings are deterministic, but if your embedding backend is behind a load balancer with slightly different model instances, results can fluctuate. Run the benchmark twice if numbers look suspicious.
